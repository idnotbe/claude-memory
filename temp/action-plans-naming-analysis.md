# Q1: 디렉토리 명칭 분석

## 프로젝트 컨텍스트

이 프로젝트(claude-memory)의 기존 디렉토리 네이밍:
- 단일 단어: `hooks/`, `assets/`, `commands/`, `skills/`, `tests/`, `temp/`, `research/`
- kebab-case: `.claude-plugin/`
- Python 자동 생성(snake_case): `__pycache__/`, `.pytest_cache/`
- **유일한 공백**: `action plans/`

**핵심 관찰:** 프로젝트의 사용자 정의 디렉토리는 100% lowercase 단일 단어 또는 kebab-case.

## 후보 평가

### Tier 1: 가장 강력한 후보

#### `plans/` -- 단순화
| 기준 | 평가 |
|------|------|
| 프로젝트 컨벤션 일관성 | **최고** -- hooks, assets, commands, skills, tests, temp, research와 동일 패턴 |
| 쉘 호환성 | **최고** -- 공백/특수문자 없음 |
| 의미 명확성 | **양호** -- "실행 계획"이라는 의미 전달. 다만 "plans"가 너무 범용적일 수 있음 |
| 간결성 | **최고** -- 6글자 |
| 충돌 위험 | git history에 `plans/` 삭제 기록 존재 (이전에 `plans/` → `action plans/`로 이름 변경한 것으로 추정). 하지만 이는 오히려 원래 의도가 plans/였음을 시사 |
| 업계 사례 | Kubernetes: `design/`, Go: `proposal/`, Swift: `proposals/` -- 단일 단어가 대세 |

**자기비판:** "plans"가 너무 일반적이라 다른 프로젝트에서 다른 의미로 사용될 수 있다는 우려가 있지만, 이 프로젝트 내에서는 맥락상 충분히 명확함. 이미 temp/, research/ 같은 일반적 이름을 쓰고 있음.

#### `action-plans/` -- kebab-case
| 기준 | 평가 |
|------|------|
| 프로젝트 컨벤션 일관성 | **양호** -- `.claude-plugin/`과 동일 패턴이지만, 대부분의 디렉토리는 단일 단어 |
| 쉘 호환성 | **양호** -- 공백 없음 |
| 의미 명확성 | **최고** -- "실행 계획"이라는 의미가 가장 명확 |
| 간결성 | **보통** -- 13글자 |
| 업계 사례 | Kubernetes `design-proposals-archive/` -- 드물지만 존재 |

**자기비판:** 의미는 가장 명확하지만, 프로젝트 내 다른 디렉토리들이 거의 모두 단일 단어인 점에서 약간 이질적. "action"이 추가하는 정보량이 "plans"만으로 부족한지 의문.

### Tier 2: 괜찮지만 차선

#### `playbooks/` -- 실행 가이드 뉘앙스
| 기준 | 평가 |
|------|------|
| 프로젝트 컨벤션 일관성 | **양호** -- 단일 단어 |
| 의미 명확성 | **양호** -- "실행 가이드라인"이라는 뉘앙스가 있으나, DevOps에서는 Ansible playbook과 혼동 가능 |
| 간결성 | **양호** -- 10글자 |

**자기비판:** 의미론적으로 매력적이지만, 이 폴더의 파일들은 실행 가이드라인보다는 "계획 문서"에 가까움. Playbook은 "이미 결정된 절차"의 느낌인데, 여기는 "앞으로 할 일의 계획".

#### `action_plans/` -- snake_case
| 기준 | 평가 |
|------|------|
| 프로젝트 컨벤션 일관성 | **낮음** -- 프로젝트에 snake_case 사용자 디렉토리 없음 (Python 자동 생성만 존재) |
| 쉘 호환성 | **양호** |
| 더블클릭 선택 | **장점** -- 전체 선택 가능 |

**자기비판:** Python 프로젝트이긴 하지만, 이 프로젝트의 디렉토리 컨벤션은 Python 패키지가 아닌 Claude Code 플러그인 구조를 따름.

### Tier 3: 부적절

#### `backlog/`, `roadmap/`, `tasks/`
- `backlog/`: Agile 용어로 "미착수 항목"의 뉘앙스 -- 진행 중/완료 파일도 있으므로 부적합
- `roadmap/`: 고수준 방향성 문서의 느낌 -- 여기는 구체적 실행 계획
- `tasks/`: `.claude/tasks/`와 충돌 가능

## 최종 판단 (내 분석)

**1위: `plans/`**
- 프로젝트 컨벤션과 100% 일치 (단일 lowercase 단어)
- git history에서 이전에 사용했던 이력 존재 (자연스러운 회귀)
- 업계 대형 프로젝트들의 패턴과 일치
- 가장 간결

**2위: `action-plans/`** (의미 명확성이 중요하다면)

**탈락:** `action_plans/` (컨벤션 불일치), `playbooks/` (의미 부적합), `backlog/` (상태 뉘앙스 부적합)
